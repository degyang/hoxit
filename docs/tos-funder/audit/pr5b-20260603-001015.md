## Verdict

PR5B direction is correct, but it should not be accepted yet. The aggregator exists and the high-level architecture is reasonable, but several contract and gating issues can cause wrong growth conclusions.

The most important issue is that BYD should trigger the negative-growth gate because latest net profit is negative, even though revenue is still positive.

## Blocking Findings

### 1. BYD Negative Growth Gate Uses AND Logic Instead Of OR

Files:

- `tos-funder/commands/tos-funder-growth.md`
- `docs/tos-funder/validation-pr5b.md`

Current behavior:

- BYD Gate 2 is marked `not_triggered`.
- Reason given: revenue is +3.46%, even though net profit is -18.97%.

Expected behavior:

- Gate 2 must trigger when either latest revenue growth or latest net profit growth is negative.
- BYD has latest net profit YoY = -18.97%, so `negative_growth.status` must be `triggered`.

Required fix:

```text
if latest_revenue_yoy < 0 OR latest_netprofit_yoy < 0:
    negative_growth.status = triggered
    confidence_cap = 60

if latest_revenue_yoy < 0 AND latest_netprofit_yoy < 0:
    confidence_cap = 50
```

BYD expected result after fix:

```text
peg_reliability = degraded
negative_growth.status = triggered
signal = neutral
confidence capped by negative growth
```

### 2. Aggregator Still Allows Raw Data Fallback

File:

- `tos-funder/commands/tos-funder-growth.md`

Current text:

```text
Determine sector_type from Fisher's output target.sector_type field, or from basicinfo query if Fisher is unavailable
```

Problem:

`/tos-funder-growth` is an aggregator. It must not issue raw hoxit/iWencai queries. It should only consume subordinate command outputs.

Required fix:

```text
Determine sector_type from Fisher output first, then Lynch output.
If both are unavailable, set sector_type="unknown", add a data_quality warning, and disable sector-specific bullish upgrades.
Do not query basicinfo from this command.
```

### 3. Schema Contract Is Ambiguous

Files:

- `tos-funder/commands/tos-funder-growth.md`
- `tos-funder/references/output-schema-examples.md`
- `tos-funder/references/skill-workflow.md`

Current behavior:

- `/tos-funder-growth` uses `matches: growth_analyst_signal`.
- It extends that schema with `consumed_signals`, `hard_gates`, and `conflict_resolution`.

Problem:

Fisher/Lynch persona signals and the aggregate signal are not the same contract. Reusing the same anchor without a discriminator makes downstream consumption ambiguous.

Preferred fix:

```text
Add output-schema-examples.md#growth_aggregate_signal.
Set /tos-funder-growth produced_schema.matches = growth_aggregate_signal.
Update skill-workflow.md and analyze.md to consume growth_aggregate_signal by default.
```

Acceptable fallback:

Keep `growth_analyst_signal`, but add:

```json
"signal_type": "growth_aggregate"
```

and explain this subtype decision in `validation-pr5b.md`.

Recommendation: use a new `growth_aggregate_signal` anchor.

### 4. Analyze Route Is Only Partially Updated

File:

- `tos-funder/commands/tos-funder-analyze.md`

Current issue:

The `family=growth` route points to `/tos-funder-growth`, but the consumed schema text still emphasizes Fisher/Lynch outputs.

Required fix:

```text
Growth: signal, strength, confidence, scoring_breakdown, consumed_signals, hard_gates, conflict_resolution — from /tos-funder-growth.
Fisher/Lynch detail outputs are optional drill-down inputs.
```

### 5. Validation Uses Assumptions As If They Were Verified Inputs

File:

- `docs/tos-funder/validation-pr5b.md`

Examples:

- BYD quant-fundamentals score = 6.0 is assumed.
- Moutai Fisher/Lynch values are described as expected.
- Ningbo Bank negative growth gate is marked as assumed not triggered.

Problem:

This is acceptable as a fixture-based validation only if the document labels inputs as fixtures. It is not acceptable as a real validation record.

Required fix:

For each sample, add:

```text
Consumed Inputs Used
- fisher: source_status = accepted_pr5a | assumed_fixture | unavailable
- lynch: source_status = accepted_pr5a | assumed_fixture | unavailable
- quant_fundamentals: source_status = accepted_pr3a | assumed_fixture | unavailable
- sentiment: source_status = accepted_pr4a | assumed_fixture | unavailable
```

Do not imply that assumed fixture values are real hoxit outputs.

### 6. BYD Confidence Derivation Is Inconsistent

File:

- `tos-funder/commands/tos-funder-growth.md`

Current issue:

The BYD sample has total score around 6.25, which implies base confidence around 63. The output says confidence is 50 and claims no caps triggered.

Required fix:

After Gate 2 is corrected, show explicit confidence derivation:

```json
"confidence_calculation": {
  "base_confidence": 63,
  "caps_applied": [
    {"gate": "negative_growth", "cap": 60},
    {"gate": "peg_reliability", "effect": "no bullish upgrade"}
  ],
  "final_confidence": 60
}
```

If final confidence is 50, explain the additional cap. Do not say "no caps triggered".

## Strategy Guidance For CC

This PR should be treated as a signal arbitration layer, not another analysis command.

Use this order:

1. Read subordinate outputs.
2. Determine data quality and hard gates.
3. Apply hard gates before weighted scoring interpretation.
4. Compute aggregate score.
5. Resolve Fisher/Lynch conflict.
6. Derive signal/strength/confidence with caps shown explicitly.
7. Output structured fields only; avoid making prose the source of truth.

Key principles:

- Aggregator must not fetch raw data.
- Degraded PEG is an opportunity note, not a bullish vote.
- Sentiment is a supporting context, not a direction driver.
- Latest-year deterioration caps conviction even when historical CAGR is strong.
- Bank low PE/PEG is not enough; require asset quality, capital adequacy, and profit growth evidence.

#### Remaining Issue: `confidence` Type Changed From int To object

Files:

- `tos-funder/commands/tos-funder-growth.md`
- `tos-funder/references/output-schema-examples.md`
- `docs/tos-funder/validation-pr5b.md`
- `tos-funder/commands/tos-funder-analyze.md`

Current schema:

```json
"confidence": {
  "base_confidence": 63,
  "caps_applied": [
    "Gate 2: latest_netprofit_yoy_pct=-18.97% < 0 → single negative cap=60"
  ],
  "final_confidence": 60
}
```

Problem:

Existing analyst signals use:

```json
"confidence": 60
```

`growth_aggregate_signal` says it extends `growth_analyst_signal`, and `growth_analyst_signal` requires `confidence: int 0-100`. Changing `confidence` to an object breaks downstream consumers that expect `signal`, `strength`, and `confidence` to be simple fields.

Recommended fix:

Keep `confidence` numeric, and move derivation into a separate field:

```json
"confidence": 60,
"confidence_calculation": {
  "base_confidence": 63,
  "caps_applied": [
    {"gate": "negative_growth", "cap": 60, "reason": "latest_netprofit_yoy_pct=-18.97%"}
  ],
  "final_confidence": 60
}
```

Then update:

- `tos-funder/commands/tos-funder-growth.md`
- `tos-funder/references/output-schema-examples.md#growth_aggregate_signal`
- `docs/tos-funder/validation-pr5b.md`
- `tos-funder/commands/tos-funder-analyze.md` if it references aggregate confidence

Validation should assert:

```text
confidence is numeric
confidence_calculation.final_confidence == confidence
```
